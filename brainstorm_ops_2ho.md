## 현황 분석 (갭 진단)

운영 담당자가 현재 대시보드를 활용할 때 **직접적으로 체감되는 정보 부족 영역**을 정리:

| 영역 | 현재 상태 | 운영에서 필요한 것 |
|------|----------|-------------------|
| **업데이트 일정** | events 배열로 하드코딩된 라인 마커만 존재 | 캘린더 뷰 + 업데이트-매출 상관 분석 |
| **수익 구조 분해** | 일별 총매출 USD 단일 라인만 표시 | 과금 채널별(가챠/패키지/패스/월정액) 매출 구분 |
| **유저 리텐션** | 점검 시점 총유저 수만 제공 (감소추세만 확인) | 코호트 기반 D1/D7/D30 리텐션 곡선 |
| **가챠 경제** | 사도/카드 가챠 총 횟수만 점검 데이터에 존재 | 가챠 종류별 풀횟수/보정/수익 기여도 |
| **지역 별 운영** | JP vs 기타 스택드 바 차트 + 과금군 이동 매트릭스 (4지역) | 지역별 DAU·ARPPU·과금율 추이 + 리전별 가챠/패키지 수요 차이 |

## 제안 1: 📅 운영 캘린더 + 업데이트 ROI 분석

### 추가 이유 (운영 관점)
현재 `events` 배열에 17개 이벤트가 하드코딩되어 라인 차트 마커로만 표시됨. 운영 담당자는 **"어떤 업데이트 구성(가챠 N종 + 패키지 M개 + 패스 유무)이 매출에 얼마나 기여했는가"**를 일목요연하게 비교해야 함. 반주년→0.75주년 스케일링 분석이 BM 섹션에 텍스트로만 존재하므로, 이를 정량적 차트로 시각화하면 **1주년 업데이트 구성 기획**에 직접 활용 가능.

### 구현 방식
- **운영 캘린더 위젯**: 가로 타임라인 형태. 각 업데이트 시점을 노드로, 노드 호버/클릭 시 해당 업데이트 상세 BM 태그(가챠 종류, 패키지 수, 패스 유무, ARPPU, 유료전환율) 툴팁 표시.
- **업데이트 ROI 스캐터 차트**: X축=과금 채널 수(가챠종수+패키지수+패스), Y축=해당 업데이트일 매출. 버블 크기=없데이트 전후 3일 합산 매출. 기준선(선형 회귀) 추가 → "채널 1개 추가당 매출 기여 α원" 추정.
- **기존 데이터 활용**: 이미 `spikes` 배열(27개 피크)과 `bmKeywords`에 가챠/패키지 정보가 있음. 이를 구조화된 JSON으로 변환만 하면 됨.

### 필요 데이터
```
update_events.json (신규)
├── date: "2026-03-26"
├── name: "반주년 이드"
├── gacha_count: 4 (사도/카드/체험/면접)
├── package_count: 20
├── has_pass: false
├── has_theme_theater: true
├── revenue_day: 508146
├── revenue_3day_sum: 1248146
├── arppu: 39.77
├── pay_rate: 9.56
└── tags: ["gacha", "package", "theme_theater", "mileage"]
```

> 💡 **구현 난이도: 낮음** — 기존 HTML 내 하드코딩된 데이터를 JSON으로 분리 + Chart.js scatter 차트 1개 추가

## 제안 2: 💰 수익 구조 분해 (Revenue Waterfall / 과금 채널별 분해)

### 추가 이유 (운영 관점)
일별 총매출 $117K~$976K의 변동이 **어떤 과금 채널의 변화에서 기인하는지** 파악 불가. 예를 들어 6/18 매출 $976K가 가챠 충전인지, 한정패키지 직구인지, 트릭컬패스 갱신인지 구분되지 않음. 운영 담당자는 **"이번 업데이트에서 가챠 수익 vs 패키지 수익 비율은?"**을 실시간으로 확인하여 다음 업데이트 패키지 구성을 조정해야 함.

### 구현 방식
- **과금 채널별 스택드 바 차트**: 각 날짜별 매출을 충전(엘리프 구매) / 가챠(사도·카드·성격모집) / 패키지(한정·스텝업·신입교주) / 구독(월정액 4종·트릭컬패스·레벨패스) / 기타 로 분해.
- **업데이트별 파이 차트**: 해당 업데이트일(±3일)의 과금 채널별 구성비. 반주년(3/26) vs 0.75주년(6/18) 비교 → 패키지 비중 변화 시각화.
- **월별 과금 채널별 트렌드**: 월 단위로 충전/가챠/패키지/구독 비율 변화 추이 (Plotly-style stacked area 또는 100% 스택드 바).

### 필요 데이터
```
revenue_breakdown.json (신규 — 현재 보유 데이터에 없음)
├── date: "2026-06-18"
├── gacha_revenue: 320000    (사도+카드+성격모집 가챠 수익)
├── package_revenue: 280000  (한정+스텝업 패키지 수익)
├── subscription_revenue: 180000 (월정액+패스 갱신)
├── charge_revenue: 150000   (엘리프 충전/과금)
├── other_revenue: 46414     (부띠끄/스킨 등)
└── total: 976414
```

> ⚠️ **데이터 가용성: 낮음** — 이 데이터는 현재 일별 매출 CSV에 없음. **DB dump에서 과금 유형별 통계 추출 필요**, 또는 게임 내 결제 로그에서 집계. 운영 담당자가 DB 쿼리로 추출 가능한지 확인 선행 필요.

## 제안 3: 🔄 코호트 리텐션 + 잔존 유저 과금 추이

### 추가 이유 (운영 관점)
점검 DB dump에서 **총유저 305K→256K로 16% 감소**하고 있으나, 이것이 이탈인지 정리인지 판단 불가. 더 중요한 것은 **"특정 업데이트로 유입한 신규 유저가 D7/D30에 얼마나 남아있고, 얼마나 과금하는가"**임. 현재 `new_accounts` 일별 데이터(1/1: 14,593 → 6/18: 1,535)가 있지만, 이 신규 유저의 행동 추적이 전무함.

운영 담당자는 **반주년 신규 코호트 vs 0.75주년 신규 코호트의 리텐션을 비교**하여, 1주년 신규 유입 BM(신입교주 패키지 가치, 첫 구매 유도 타이밍)을 설계해야 함.

### 구현 방식
- **코호트 히트맵**: 행=가입 주(예: W1 1/1~1/7, W2 1/8~1/14...), 열=D0/D1/D3/D7/D14/D30. 색상=잔존율(%). 셀 클릭 시 해당 코호트의 과금률/ARPPU 표시.
- **코호트별 누적 과금 곡선**: 주요 코호트(신정, 반주년, 0.75주년)를 선택하여, 가입 후 N일까지 누적 1인당 과금액 추이 라인 차트.
- **D7 리텐션 트렌드 라인**: 주간 단위 D7 리텐션률 변화 → 업데이트 전후 리텐션 변동 패턴 분석.

### 필요 데이터
```
cohort_retention.json (신규 — DB 쿼리 필요)
├── cohort_week: "2026-W13"  (반주년 주간)
├── cohort_size: 8500        (해당 주간 가입자)
├── retention:
│   ├── d0: 100%, d1: 72.3%, d3: 48.1%, d7: 31.2%
│   ├── d14: 22.5%, d30: 16.8%
│   └── d60: null (아직 도달 안 함)
├── conversion_d7: 12.4%    (D7 내 첫 과금 비율)
├── cumulative_arppu_d30: $8.50  (30일 누적 1인당 과금)
└── region_split: { jp: 78%, sgp: 8%, va: 6%, other: 8% }
```

> ⚠️ **데이터 가용성: 중간** — 점검 DB dump에서는 일별 신규수만 확인 가능. **실제 코호트 추적은 서버사이드 로그 또는 GA 기반 필요**. 대안으로 일별 신규 수 × 다음날 DAU 증감분으로 간이 D1 리텐션 추정 가능.

## 제안 4: 📋 업데이트 실행 체크리스트 + 일정 Gantt

### 추가 이유 (운영 관점)
현재 대시보드는 "분석 결과물"만 보여주고, **"앞으로 해야 할 일"의 관리 기능**이 없음. ✅ 실행 과제 섹션이 있지만 정적 텍스트 리스트일 뿐, 담당자/마감일/진행상태 추적이 불가. 운영 담당자는 **반/0.75주년의 성공 패턴을 매뉴얼화**하여 1주년 업데이트 실행 계획(Gantt)으로 연결해야 함.

### 구현 방식
- **런웨이 카운트다운 위젯**: 현재 시점 기준 다음 대형 업데이트까지 D-N 표시. 상단 KPI 바 아래에 배치.
- **실행 체크리스트 (인터랙티브)**: 각 액션 아이템에 상태 토글(미완료/진행중/완료), 담당자, 마감일 필드. localStorage에 저장하여 브라우저에서 상태 유지.
- **간트 뷰**: 차기 업데이트(예: 1주년)의 태스크별 타임라인. DB dump 주기, 패키지 기획 완료, 사도픽업 리소스 확정, 번역 완료, QA 완료 등 마일스톤.
- **기존 데이터 활용**: 기존 `actionItems` 섹션(6개 항목)을 구조화하여 체크리스트로 변환.

### 필요 데이터
```
milestones.json (신규 — 운영 팀 내부 관리 데이터)
├── next_major_update:
│   ├── name: "1주년"
│   ├── target_date: "2026-09-26"
│   └── countdown_days: 73
├── checklist:
│   ├── { id: "C1", task: "가챠 12종 리소스 확정", assignee: "2호", due: "2026-08-15", status: "pending" }
│   ├── { id: "C2", task: "한정패키지 35개 가격 설정", assignee: "2호", due: "2026-08-30", status: "pending" }
│   └── ...
└── gantt_phases:
    ├── { phase: "기획", start: "2026-07-01", end: "2026-08-15" }
    ├── { phase: "리소스 제작", start: "2026-08-01", end: "2026-09-10" }
    └── { phase: "QA / 번역", start: "2026-09-11", end: "2026-09-23" }
```

> 💡 **구현 난이도: 중간** — localStorage 기반 체크리스트는 순수 프론트엔드로 가능. Gantt 뷰는 Chart.js가 아닌 별도 라이브러리(dhtmlxGantt 오픈소스 또는 커스텀 SVG) 필요. GitHub Pages 정적 제약 내에서는 JSON 파일을 fetch하는 방식.

## 제안 5: 🌏 지역별 운영 대시보드 (Region Deep Dive)

### 추가 이유 (운영 관점)
현재 JP vs 기타 스택드 바 차트와 4지역 과금군 이동 매트릭스만 존재. 운영 담당자는 **"JP 이외 리전에서 매출을 끌어올리려면 무엇을 해야 하는가?"**를 구체적으로 답해야 함. 예를 들어:
- SGP 유저의 ARPPU는 JP 대비 몇 %인가?
- VA 유저의 유료전환율 하락 원인은 무엇인가?
- KR/TW/HK 타겟 한정패키지를 출시하면 예상 수익은?

### 구현 방식
- **지역별 KPI 카드 행**: ALL / JP / SGP / VA 각각에 대해 DAU, 유료자수, 유료전환율, ARPPU, 총매출을 카드로 표시. JP와의 상대적 비율(%)도 함께 표시.
- **지역별 과금 구조 레이더 차트**: 각 지역별 과금군 분포(무과금~고래 비율)를 레이더/폴라 차트로 비교 → "JP는 고래 비율이 높고 SGP는 소과금 비중이 높다" 같은 패턴 시각화.
- **지역별 업데이트 반응 히트맵**: 행=주요 업데이트(반주년, 0.75주년 등), 열=지역(JP/SGP/VA). 셀=해당 업데이트 전후 DAU/매출 변화율. → "SGP에서는 이벤트 반응이 JP보다 2주 지연된다" 같은 인사이트.
- **기존 데이터 활용**: `region_transition.json`에 4지역 과금군 매트릭스 이미 존재. 점검 DB에서 지역별 유저 수 추출 가능 (inspection_metrics.json의 total_users를 리전별로 분해 필요).

### 필요 데이터
```
region_daily_metrics.json (신규)
├── date: "2026-06-18"
├── regions:
│   ├── jp: { dau: 98000, paid: 15000, pay_rate: 15.3%, revenue_usd: 820000, arppu: 54.67 }
│   ├── sgp: { dau: 8500, paid: 1800, pay_rate: 21.2%, revenue_usd: 45000, arppu: 25.00 }
│   ├── va: { dau: 12000, paid: 1500, pay_rate: 12.5%, revenue_usd: 35000, arppu: 23.33 }
│   ├── kr: { dau: 3000, paid: 400, pay_rate: 13.3%, revenue_usd: 8000, arppu: 20.00 }
│   └── other: { dau: 4328, paid: 751, pay_rate: 17.3%, revenue_usd: 68414, arppu: 91.10 }
└── total: { ... }
```

> ⚠️ **데이터 가용성: 중간** — JP 일별 매출 CSV는 이미 존재. SGP/VA 과금군 데이터도 있음. **다만 일별 지역별 DAU/과금률 데이터는 추가 추출 필요**. 점검 DB dump에 리전 필드가 있다면 추출 가능성 있음.

### 데이터 파이프라인 구축 후 실행
3. **지역별 DAU/과금률**: 점검 DB에서 리전 필드 기반 일별 추출 → `region_daily_metrics.json` 자동 생성
4. **코호트 리텐션**: 서버사이드 유저 행동 로그에서 코호트 집계 파이프라인 구축
5. **수익 구조 분해**: 결제 로그에서 과금 유형(가챠/패키지/구독) 분류 → 일별 분해 JSON 생성

## 🎯 결론

운영 담당자 관점에서 **가장 시급한 것은 "분석 결과물 → 실행 계획 → 진행 추적"의 흐름 연결**이다. 현재 대시보드는 훌륭한 분석 도구지만, **"다음에 무엇을 언제까지 해야 하는가?"**에 대한 답을 주지 못한다.

1~2번 제안(캘린더+ROI, 체크리스트)은 **기존 데이터와 순수 프론트엔드만으로 즉시 구현 가능**하며, 운영 담당자의 실제 업무 효율을 가장 직접적으로 높일 수 있다. 3~5번 제안은 데이터 파이프라인 확충 후 도입을 권장.
